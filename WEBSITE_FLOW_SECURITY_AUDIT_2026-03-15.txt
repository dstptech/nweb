2) ANSWER TO MANAGER QUESTION: PASSWORD VISIBLE IN URL?
================================================================================

Question: "When entering username/password, URL shows password. Is this true?"

Findings from current code:
- In this repository, implemented login UI is currently missing/empty.
- I found no active frontend code sending credentials via query string.
- Backend login endpoint expects POST body (request.data), not query params.

So in current committed code:
- I cannot confirm an active password-in-URL bug because login page is not implemented.

When this issue happens generally:
- It happens if frontend submits login with GET (query params), or via axios/fetch params instead of request body.

Required fix (must do when implementing login page):
1. Submit login via POST only.
2. Send credentials in JSON body:
   { "email": "...", "password": "..." }
3. Never send credentials in URL/query string.
4. Use HTTPS in all non-local environments.
5. Do not log full query strings containing secrets.

3) CRITICAL ERRORS + SECURITY ISSUES TO FIX
================================================================================

3.1 CRITICAL: Blog API crashes due missing model field
- File: dstp_backend/apps/blog/views.py line 54
- Code filters on is_published, but Blog model has no is_published field.
- Result: list/retrieve behavior can fail with FieldError.
- Fix:
  - Option A: add is_published BooleanField + migration.
  - Option B: remove filter and use another publish strategy.

3.2 HIGH: Service search fields reference non-existent columns
- File: dstp_backend/apps/services/views.py lines 117, 176
- Service model uses title/feature but view searches name/description for models that do not have name.
- Result: search queries can crash.
- Fix:
  - ServiceViewSet search_fields -> ["title", "description"]
  - ServiceFeatureViewSet search_fields -> ["feature"]
  - Ordering fields should also match real columns.

3.3 HIGH: Refresh token flow bypasses configured token rotation policy
- Config says rotation+blacklist enabled:
  - dstp_backend/config/settings/base.py lines 237-238
- But custom refresh code just creates access token directly from refresh token:
  - dstp_backend/apps/authentication/services.py lines 167-189
- Result: refresh rotation policy is effectively bypassed.
- Fix:
  - Use standard SimpleJWT TokenRefreshView/serializer flow (rotated refresh handling).

3.4 HIGH: Pending migration for password_last_changed
- Field exists in model:
  - dstp_backend/apps/authentication/models.py line 60
- makemigrations --check shows new migration required.
- Risk: schema drift between code and DB.
- Fix:
  - Create/apply migration for authentication customuser field.

3.5 HIGH: Auth/login brute-force protection not properly scoped
- base.py defines throttle rate for "auth" but DEFAULT_THROTTLE_CLASSES only has AnonRateThrottle/UserRateThrottle.
- Scoped "auth" limit is not actually applied.
- Fix:
  - Add ScopedRateThrottle + throttle_scope="auth" on login/register/password reset views.

3.6 HIGH: Visitor cookie not Secure in middleware
- File: dstp_backend/middleware/visitor_tracking_middleware.py line 61
- secure flag is commented out.
- Risk: cookie can travel over HTTP if deployed incorrectly.
- Fix:
  - Set secure=True in production path; use env-driven flag.

3.7 MEDIUM: Visitor stats endpoint exposes IP/device info to AllowAny
- File: dstp_backend/apps/core/views.py lines 137 and 171
- Any requester with cookie can fetch sensitive visitor metadata.
- Fix:
  - Restrict access, minimize returned PII, or remove IP from response.

3.8 MEDIUM: Duplicate URL includes in config
- File: dstp_backend/config/urls.py lines 29-37
- Duplicated include patterns for auth/blog/projects/careers/etc.
- Risk: confusion, duplicate schema paths, maintenance mistakes.
- Fix:
  - Keep only one include path strategy.

3.9 MEDIUM: Environment loading inconsistency for DB credentials
- base.py reads SECRET_KEY via decouple config(), but DB vars via os.environ.get(...).
- If env vars are not exported, DB config resolves incorrectly.
- Fix:
  - Use config(...) consistently for DB values OR ensure env export strategy is enforced.

3.10 MEDIUM: LocalStorage token storage in admin frontend
- File: dstp_frontend/admin/src/api/axiosInstance.js lines 15-17
- Risk: token theft on XSS.
- Fix options:
  - Prefer HttpOnly secure cookie strategy for refresh tokens.
  - Harden CSP and XSS defenses if localStorage retained.

3.11 MEDIUM: Admin logout not fully implemented
- TopBar uses alert("Logged out!") only.
- Sidebar removes local storage but does not call backend logout blacklisting.
- Fix:
  - Always call POST /api/v1/auth/logout/ with refresh token and then clear client state.

3.12 QUALITY RISK: Test suite mostly empty
- Multiple backend auth/integration test files are empty.
- Risk: regressions/security breaks go undetected.
- Fix:
  - Add auth, permissions, and endpoint contract tests first.

4) SCHEMA MISMATCHES (FRONTEND EXPECTS vs BACKEND PROVIDES)
================================================================================

4.1 Homepage Editor mismatch (major)
Frontend expects structured sections:
- about, services, tech_stack, projects, testimonials, contact
Backend Homepage model currently has only:
- hero_title, hero_subtitle, hero_image, banner_text, banner_image, stats_title, stats_value

Impact:
- Homepage editor UI cannot persist most tabs.
- Current code already comments out non-hero payload fields.

Fix choices:
A) Extend Homepage schema with JSONField columns for these sections.
B) Normalize into dedicated tables (recommended for scale and queryability).

4.2 Services admin table mismatch
Admin Services UI fields:
- name, category, status, created
Backend Service model fields:
- category(FK), title, description, is_featured

Impact:
- UI cannot map directly without adapter/field updates.

Fix:
- Align frontend form/table to backend shape OR evolve backend schema to include needed fields.

4.3 Team management mismatch
Admin Teams UI expects:
- role, department, status, joined, CRUD endpoint
Backend currently only has CustomUser basic fields + role/date_joined.
No dedicated Team member API exists.

Fix:
- Decide if team page manages CustomUser or separate TeamProfile model.
- Build REST endpoints accordingly.

4.4 Blog publish status mismatch
Frontend/admin flows usually need draft/published control.
Backend Blog model has no publish status field, but view assumes one.

Fix:
- Add status/is_published + optional published_at.
